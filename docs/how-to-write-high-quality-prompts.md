# 如何编写高质量提示词

## 一、核心原则

### 1. 清晰 > 巧妙
提示词不是猜谜游戏。直接说你想要什么，比用含蓄的方式描述更有效。

**反面：** "能不能帮我看看这段代码有没有什么不对的地方"
**正面：** "找出以下代码中的 bug，列出每个 bug 的：位置、原因、修复建议"

### 2. 结构 > 篇幅
长提示词不等于好提示词，但结构化的提示词几乎总是优于无结构的。

### 3. 示例 > 描述
当输出格式或行为难以用文字精确描述时，给一个具体的示例胜过千言万语。

---

## 二、结构化模板

一个高质量提示词通常包含以下要素（按需取舍，不必全部包含）：

```
# 角色
你是一个 [具体角色]，擅长 [具体领域]。

# 任务
[清晰描述你要模型做什么]

# 上下文
[提供必要的背景信息]

# 约束
- [约束条件 1]
- [约束条件 2]

# 输入
[具体的输入数据]

# 输出格式
[期望的输出结构或格式]

# 示例
输入：xxx
输出：xxx
```

## 三、Markdown 格式为什么有效

### 3.1 层级分明

```markdown
# 一级标题 → 角色或任务总述
## 二级标题 → 模块划分
### 三级标题 → 子模块
```

模型对标题层级有天然的理解能力，能据此判断指令的主次关系。

### 3.2 边界清晰

用代码块、分割线、引用块明确划分不同区域：

- **代码块** `` ``` `` — 隔离输入数据、示例代码、模板
- **分割线** `---` — 分离章节
- **引用块** `>` — 标注参考材料或约束来源

### 3.3 列表天然适配指令

```markdown
- 无序列表 → 并列条件，无优先级
1. 有序列表 → 步骤流程，有先后顺序
- [ ] 任务列表 → 待办或检查清单
```

### 3.4 表格处理结构化数据

```markdown
| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| name | string | 是 | 用户名 |
```

比用纯文字描述字段关系要清晰得多。

## 四、实战技巧

### 4.1 用分隔符隔离输入

当提示词包含用户输入或外部数据时，必须用分隔符防止指令注入和混淆：

```markdown
请对以下文本进行摘要：

<text>
{{用户输入的文本}}
</text>
```

常用分隔符：
- `"""` 或 `'''` — 简单场景
- `<tag>...</tag>` — XML 标签，语义清晰
- `` ``` `` — 代码或结构化数据

### 4.2 指定思考路径

让模型先分析再输出，可以显著提升复杂任务的准确率：

```markdown
请按以下步骤分析：
1. 先识别问题类型
2. 再列出已知条件
3. 然后给出推理过程
4. 最后给出结论
```

或者使用 chain-of-thought：

```markdown
在给出最终答案前，先展示你的推理过程。
```

### 4.3 给出负面约束

告诉模型"不要做什么"和"要做什么"同样重要：

```markdown
# 要求
- 使用中文回答
- 代码使用 TypeScript

# 不要
- 不要解释基础概念
- 不要使用 any 类型
- 不要添加超出要求的额外功能
```

### 4.4 用示例锚定输出格式

模糊描述：

```markdown
输出一个 JSON，包含用户信息。
```

精确示例：

```markdown
请按以下格式输出：

```json
{
  "name": "张三",
  "age": 28,
  "skills": ["TypeScript", "React"]
}
```
```

### 4.5 处理长上下文时用摘要+细节

```markdown
# 背景
项目是一个微信小程序，用于培训机构的学生管理。

# 当前任务详情
（这里展开具体描述……）
```

先给概览，再给细节，模型能更好地分配注意力。

## 五、常见反模式

| 反模式 | 问题 | 改进 |
|--------|------|------|
| 把所有要求写成一整段 | 模型难以区分主次 | 用标题和列表拆分 |
| 模糊的量化词："适当"、"一些" | 模型理解不一致 | 用具体数字或范围 |
| 只描述不举例 | 输出格式不可控 | 给出 input/output 示例 |
| 忽略边界情况 | 模型自行发挥 | 明确说明异常处理方式 |
| 嵌套过深的条件 | 模型容易遗漏 | 扁平化，用 if-then 表格 |
| 角色设定太空泛 | 无实际引导作用 | 角色要具体到专业领域+行为特征 |

## 六、提示词迭代流程

```
编写初版 → 测试 → 分析问题 → 修改 → 再测试 → 稳定
```

关键复盘问题：

1. **输出是否稳定？** — 同一提示词多次运行，结果差异大吗？
2. **边界情况是否覆盖？** — 输入为空、超长、格式异常时表现如何？
3. **指令是否被完整遵循？** — 有没有"遗忘"某些要求？
4. **能否更简洁？** — 删掉某句话后效果是否变差？如果没有，删掉它。
